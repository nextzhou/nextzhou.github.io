低精度浮点数与监控数据收集

发表于 2020-01-01 更新于 2020-05-31

做后端的日子里几乎每天都会与监控打交道，监控系统能帮我们快速地了解服务当前与历史的整体状态。
一套完整的监控系统，一般由 数据采集、数据存储(时序数据库)、数据查询与展示、异常报警 等环节组成。
今天，我们以 Prometheus 为例，来聊一聊数据采集环节中的难点与解决方案。

Histogram & Summary

熟悉 Prometheus 的小伙伴应该知道，在 Prometheus 中， Metrics 数据分为以下四类:

Metric 简介 案例
Counter 记数器 QPS
Gauge 标量 CPU 占用率
Histogram 直方图 耗时统计
Summary 摘要 耗时统计

各 Metric 详细含义请查看官方文档，在此不再赘述

可以看到，上表中 Histogram 与 Summary 我都使用了“耗时统计”这个案例。实际上，它俩的目的是相同的：“收集标量样本，并提供统计数据”。但区别在于它们出统计结果的方式不同。

Summary

Summary 在 Client 端便会将样本聚合成统计结果，当 Prometheus 过来拉取数据时，是没有原样本信息的。

好处是:

  • 输出数据量少
  • 查询速度快

缺点是:

  • 消耗 Client 端性能
  • 由于原样本信息丢失，查询时缺乏灵活性

这个缺乏灵活性具体是指什么呢？例如我一个服务部署了 $S_1$、$S_2$ 两个节点，通过 Summary 监控了处理一次请求的耗时。那么我可以查看到 $avg(S_1)$ 与 $avg(S_2)$，但我却无法查看 $avg(S_1,S_2)$ ，因为 $avg(S_1,S_2) \ne avg((avg(S_1)+avg(S_2)))$。关于这一点，在官方文档 Prometheus 最佳实践也有提到过。

Histogram

Histogram 在 Client 端也是会聚合数据的，但它的聚合不像 Summary 那么彻底。在新建一个 Histogram 时需要传入一个 Buckets 配置，不传的话就使用 默认值。通过这份配置，Histogram 将 float64 整个数域划分成len(Buckets)+1 个大粗桶，然后把所有的样本数据划分到这几个桶里来，最终的结果就是“Buckets 划分以及各自对应的样本数量”。

好处是：

  • 对 Client 端的性能消耗很少
  • 查询灵活

缺点是：

  • 输出数据量大
  • 查询速度慢
  • 需要规划 Buckets 划分

查看 Golang 相关文档可知在 Go 中 float64 使用的是 IEEE 754标准，其数域范围大约是从 $-1.7977e+308$ 到 $1.7977e+308$。而且分布也是不均匀的，越靠近0越密集。这种情况下，想要设计出一个通用的 float64 Buckets 几乎是不可能的，但好在我们是针对监控场景做优化，那让我们来看看在监控场景下的数据有什么特别吧：

精度要求不高

我想应该不会有人关心某请求的耗时到底是 123.456ms 还是 123.457ms 吧？对于监控数据来说，1% 以内的误差无关痛痒。而在 IEEE 754 标准中，float64 使用了整整 52 bits 来保存精度信息！这对于监控数据来说简直是种不可容忍的浪费。

数据分布相对集中

虽说 float64 的数域很大，但我们监控的数据往往不会跨很多数量级。一个平均100ms的请求，不会出现 1ns 完成，也不会出现1年完成。一个1KB的包，也不可能某次变成1TB(PS: 就算出现了，这数量级相比 float64 仍然是小case)。

如何设计低精度浮点数

根据上节监控数据的特点以及 IEEE 754标准，我们可以推导出在内存中，一个 float64 的 64 bits 中哪些是重要的，以及其分布会有什么特点：

sign

sign 是浮点数的符号位，空间占用是 1 bit, 正数还是负数就是由它来决定的。很显然，这 1 bit 是必不可少的，虽说我上面举的例子都是正数，但也可能会有负数的出现。例如统计两个相互无依赖的事件发生的时间差。

exponent

exponent 是 base 2 的指数位，在 float64 中，它占用 11 bits。要注意的一个细节是，它比实际指数要小 1023，这样做的目的是为了表示负指数。或者换个思路，你把它看做 11 bits 的带符号整数就可以了。

虽然我们能接受丢失精度，但缩小数域范围却是不能接受的，这会导致只有部分 float64 可以成功转换成我们的低精度浮点数，从而带来很多数据正确性上的隐患。因此，这 11 bits 也是必不可少。

但是，我们上面有提到监控数据分布相对集中、跨数量级较少。这意味这什么呢？这意味着 exponent 的实际取值是极其有限的。 还是拿耗时举例子：最小耗时 1us，最大耗时24h。这应该已经覆盖了绝大多数请求耗时的范围了吧？两者换算比例 $1:8.64e10$，也就是在十进制下才跨不到 11 个数量级，换算成二进制就是跨了不到 37 个数量级(PS: 由于 $2^{10} = 1024 \cong 10^3$, 因此十进制换二进制数量级大概是 $\times 3$)。也就是说，在这种较为极端的 Case 下， exponent 也只有最多 37 个取值，相比所有取值 $2^{11} = 2048$ 种取值实在是小了太多。

fraction

fraction 用来保存数据的精度信息，在 float64 中占据了整整 52 bits！！！对于监控数据而言，我们只需要关心其中的一小部分就足够了。为了简化问题，我最终决定在 fraction 中取 8 bits 来保存精度信息，正好是 1 byte 的大小。 1/256 的精度，对于监控场景来说绰绰有余了。
